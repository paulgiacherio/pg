Codex skills are quietly replacing the need for a CMS on simple sites

March 23, 2026

For a personal site, a traditional CMS often solves a problem and creates three more. It gives you an editor and a publishing interface, but it also introduces overhead, abstraction, and a layer between the author and the site itself. If the site is small, that trade can start to feel silly very quickly.

What has felt more useful to me is the idea of a workflow as the CMS. The content still lives in plain HTML and CSS. The repo still carries the history. But the repeatable publishing logic gets pulled into a skill, where it can remember the exact files to touch, how a changelog should be structured, how versioning should work, and even how album art should be fetched from a music link.

That distinction matters. I do not really want a dashboard for this site. I want the site itself to remain legible. I want to be able to open the files, understand them, and move them by hand if I need to. But I also want some of the convenience people usually associate with publishing tools: version updates, canonical changelog records, syncing a homepage teaser, or keeping the footer music aligned with the newest release.

Skills are a nice middle path because they can encode decisions without hiding the site. They can say: this is where the current version lives, this is how summaries are written, this is what counts as a patch versus a minor release, and this is how music metadata should be attached to an entry. Once those rules exist, the site starts behaving like it has an editorial system even though nothing about it has become bloated.
